# Instances + Object Diagrams Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
> **Gate:** full
> **Rigor:** tdd-per-task

**Slug:** `instances-object-diagrams` (used in the `Plan:`/`Plan-Tasks:` commit trailers).

**Goal:** Land the whole data model + conformance validation for UML `InstanceSpecification` (objects) and links (object-diagram edges) — model, grammar, generated bindings, and warn-only validation — so a later object-diagram renderer and slice 4 (instances-as-lifelines) build on a stable shape. **No renderer.**

**Architecture:** Objects are one more element kind in the same pool as classes (design spec §5). A standalone `type: uml.InstanceSpecification` doc, and an inline `## Members` object in a Diagram, both produce a pool `InstanceSpecification` `Node` carrying `slots: Vec<Slot>` (values, not attributes). Its classifier is an `InstanceOf` edge; an object-to-object link is a `Links` edge — both ordinary `Model.edges`. Two new controlled-prose productions (`## Slots`, the inline `instance of … with … set to …` member, and the `instance of` / `links` relationship verbs) parse deterministically and round-trip byte-identically. Three warn-only diagnostics check slot/classifier conformance. The Rust runtime shape is emitted to TypeScript through Tsify, so the wasm binding regenerates and the TS consumers carry `slots` additively.

**Tech Stack:** Rust (`waml` crate; `cargo`, serde, `tsify_next`/`wasm-bindgen` codegen, `regex`), TypeScript monorepo (`packages/`, `pnpm`, Vitest).

## Global Constraints

- **Full CI gate, in this order:** `cargo fmt --check` → `cargo clippy` (no new warnings) → `cargo test --workspace` → `pnpm build:wasm` → `pnpm lint` → `pnpm build` → `pnpm -r test`. Every task must leave the tree green under the full gate before its commit.
- **`is_classifier()` is explicit — no `_ =>` catch-all.** `InstanceSpecification` → `false` is an added match arm. `is_view()` stays `false` for it.
- **`Slot` mirrors `Attribute`:** serde `ref` (skip-if-none) + tsify. `Node.slots` is skip-if-empty (mirrors `values`).
- **`RelationshipKind::{InstanceOf, Links}` are ordinary edges in `Model.edges`** — no separate pool/type. `Links` reuses `Edge.name: Option<AssocName>` for its Association; neither takes ends.
- **Controlled-prose reserved tokens:** `instance of`, `with`, `set to`, `and`, `as`, `links`. A slot value is a quoted string, a bare ident/number, or a `[Label](./ref.md)` link; a bare value may not contain a reserved word (quote it). **Canonical serialize joins slots with ` and `** so parse → serialize is byte-identical.
- **Validation is warn-only** (three codes). Object diagrams are tolerant, like `Unknown` types (OKF §9).
- **Out of scope (guards):** NO renderer. Do NOT touch `Lifeline.ref_` (stays classifier-only; instances-as-lifelines is slice §7.4). Inline is **instances-only** (not arbitrary elements). Do NOT redesign or break round-trip of any untouched syntax layer. Do NOT modify the TS `RELATIONSHIP_KINDS` / `ENDED_KINDS` lists or `packages/okf/src/grammar.ts` (the class-relationship authoring UI is a separate future concern). Do NOT hand-edit the generated `packages/wasm/src/generated/*` files — they are regenerated by `pnpm build:wasm`.

## Notes / risks

- **Two binding regens (Task 2, Task 6) — by design.** Task 2 introduces the tsify-visible runtime types (`Slot`, `Node.slots`, `RelationshipKind` variants) that Tasks 3–5 populate without further shape change; Task 6 introduces the `DiagCode` variants. Both bundle Rust + regen in one green commit. Tasks 3/4/5 assert `git diff --exit-code` shows NO binding change.
- **Round-trip is a hard requirement.** Slot surface forms (quoted vs bare vs link) and the inline `with … set to … and …` clause must serialize byte-identically; the syntax layer carries the surface form (`SlotValue`), distinct from the resolved runtime `Slot`.
- **`build_flows` is the promotion template.** Inline instances pool at `{diagram}#name` and are auto-added to the diagram's members exactly as activity nodes pool at `{behavior}#id` under `FlowDoc.nodes`.
- **`MemberGroup.members` type change ripples** to `resolve_group`, `check_group_members`, `render_members_block`, and any test constructing members — all compile-forced, mechanical. `push_line_errors`/`push_group_errors` are generic over `Line<T>` and need no change.
- **`pnpm build:wasm` needs the wasm toolchain** (`wasm-pack`, `wasm32-unknown-unknown`) — part of the CI gate as for every change here.

## Cross-slice contract

- Instances are **pooled** (design spec §5) — `InstanceSpecification` is a `Model.nodes` member for which `is_classifier()` is `false`. This is consistent with slice-1's single classifier predicate: instances live in the pool but are not classifiers.
- `InstanceOf` / `Links` are **structural `Model.edges`**, distinct from behavior's `flow_edges` pool and from interaction-local `SeqEdge`s. No shared-pool collision.
- `Lifeline.ref_` stays classifier-only; widening it to instances is slice §7.4, explicitly deferred. This slice adds NO sequence handling.
